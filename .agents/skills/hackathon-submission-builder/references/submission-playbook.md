# Submission playbook

## Requirement inventory

Build a table with these columns before drafting:

| ID | Source | Exact requirement | Format/limit | Evidence | Final artifact | Status |
| --- | --- | --- | --- | --- | --- | --- |

Distinguish `required`, `judging advantage`, and `optional`. Preserve URLs and access dates in the working notes. When rules conflict, prefer the most recent official source and flag the conflict.

## Baseline artifact set

Create only what is relevant, but explicitly consider:

- form-ready title, tagline, short/long description, track choice, team, links, and technology answers
- problem, user, insight, solution, differentiation, impact, and next steps
- public demo and repository links, or reproducible local setup when public hosting is not required
- README with prerequisites, exact commands, fixture/demo mode, architecture, testing, and limitations
- architecture diagram and implementation explanation tied to actual code
- AI/model/Codex usage disclosure, including required session IDs supplied by the user or available evidence
- license, third-party attribution, data provenance, privacy/security notes, and rule-specific declarations
- pitch deck or script, screenshots, thumbnail, demo MP4, captions, and a timed narration script
- build/test output and a criterion-by-criterion evidence index

## Evidence rules

- Derive product and implementation claims from source, tests, git history, deployment output, or user-provided facts.
- Label estimates and future plans. Never turn them into shipped features.
- Do not infer user counts, revenue, performance, awards, partner approval, originality, or eligibility.
- Capture commands, dates, and versions for reproducibility. Redact secrets from logs.
- Prefer a small number of strong screenshots and proof points over repeated marketing images.

## Form copy

Write copy at every requested character limit. Keep a canonical claims sheet so the form, README, deck, and narration do not contradict one another. Put URLs on their own labeled lines for easy pasting. Replace empty fields with `TODO-BLOCKED:` rather than plausible-looking filler.

## Packaging

Use a new staging directory. Declare every copied path in the package spec. Review the generated manifest and status report, then archive only if the competition asks for an archive. Keep editable sources alongside rendered PDFs or images when useful to the user, but submit only allowed formats.
