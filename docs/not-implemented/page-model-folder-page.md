# 대안 B: 폴더와 페이지 경계

## 요약

전통적인 폴더/페이지 구조를 블록 트리 위에 명시한다. 페이지로 지정된 블록은 탐색 트리의 말단이며, 그 아래의 모든 자손 블록은 해당 페이지의 콘텐츠가 된다.

페이지 안에는 다른 페이지나 폴더를 중첩하지 않는다.

## 구조

```text
Product Studio                  폴더
├─ Projects                    폴더
│  ├─ Build Week Launch        페이지
│  │  ├─ Product thesis        콘텐츠 블록
│  │  ├─ Core experience       콘텐츠 블록
│  │  │  └─ Three views        콘텐츠 블록
│  │  └─ Launch checklist      콘텐츠 블록
│  └─ Mobile redesign          페이지
└─ Research                    폴더
   └─ User interviews          페이지
      ├─ Interview 01          콘텐츠 블록
      └─ Findings              콘텐츠 블록
```

페이지보다 위쪽은 탐색 구조이고, 페이지 아래쪽은 문서 콘텐츠 구조다. 한 경로에는 최대 하나의 페이지 경계만 존재한다.

## 핵심 규칙

- 루트는 항상 폴더다.
- 폴더는 폴더 또는 페이지를 자식으로 가질 수 있다.
- 페이지는 콘텐츠 블록만 자식으로 가질 수 있다.
- 콘텐츠 블록은 콘텐츠 블록을 재귀적으로 가질 수 있다.
- 페이지 또는 콘텐츠 블록 아래에는 새 폴더나 페이지를 만들 수 없다.
- 페이지를 선택하면 모든 정본 콘텐츠 자손을 끝까지 문서로 렌더링한다.
- 콘텐츠 블록을 직접 열면 소유 페이지를 열고 해당 블록으로 스크롤하고 강조한다.
- 폴더를 선택하면 전체 자손 문서를 펼치지 않고 직계 폴더와 페이지의 인덱스를 보여준다.

## 왼쪽 트리와 오른쪽 문서

왼쪽 트리에는 폴더와 페이지만 표시한다. 페이지 콘텐츠 블록은 문서 안에서 편집하고, 검색이나 참조로 직접 접근할 수 있다.

오른쪽 작업면은 선택 대상에 따라 두 형태가 된다.

- 폴더 선택: 하위 폴더와 페이지의 이름, 요약, 하위 페이지 수와 진행률을 보여주는 인덱스
- 페이지 선택: 페이지 아래의 모든 정본 콘텐츠 블록을 렌더링하는 문서

페이지 콘텐츠는 임의의 깊이에서 잘리지 않으므로 문서 안에 구조 탐색용 펼침 아이콘이나 `더 보기` 행이 필요하지 않다. 큰 페이지의 성능은 가상화와 지연 렌더링으로 해결한다.

## 데이터 모델

별도 `page`나 `directory` 테이블을 만들지 않고 블록에 페이지 경계만 저장할 수 있다.

```text
block(
  id, parent_id, path, rank?, body_md, status?,
  is_page, ...
)
```

블록의 역할은 `is_page`와 조상 관계로 파생한다.

```text
is_page == true                 -> page
is_page인 조상이 있음           -> content
그 외                           -> directory
```

`is_page`는 정본 순서를 나타내는 `rank`와 독립적이다. 내부 이름을 `document_root`로 두면 이 값이 별도 엔티티가 아니라 문서 콘텐츠 영역의 시작점임을 더 분명히 할 수 있다.

## 구조 제약과 연산

- `is_page`인 조상을 가진 블록에는 `is_page = true`를 설정할 수 없다.
- 폴더나 페이지를 페이지 콘텐츠 아래로 이동할 수 없다.
- 페이지 콘텐츠를 폴더 영역으로 이동하면 역할이 콘텐츠에서 폴더로 바뀔 수 있다.
- 페이지를 폴더로 바꾸면 기존 콘텐츠 자손의 역할이 탐색 구조로 바뀔 수 있다.
- 역할이 바뀌는 이동과 전환은 사전 확인 또는 명시적인 변환 UX가 필요하다.
- 페이지 지정과 해제를 지원한다면 기존 연산 외에 속성 변경 계약이 추가로 필요하다.

## 장점

- 폴더, 페이지와 페이지 본문이라는 익숙하고 예측 가능한 멘탈 모델을 제공한다.
- 루트를 선택해도 모든 콘텐츠를 한 문서에 렌더링하지 않는다.
- 왼쪽 탐색 트리가 문단이나 작업 블록으로 과밀해지지 않는다.
- 페이지 문서는 숨겨진 자손 없이 완전하게 표시할 수 있다.
- 문서 안의 펼침 깊이, 이어보기와 펼침 아이콘 규칙이 필요 없다.

## 위험과 비용

- 폴더, 페이지와 콘텐츠라는 역할이 생겨 기존의 단일 블록 온톨로지가 약해진다.
- 콘텐츠 블록을 독립된 페이지처럼 바로 열 수 없고 소유 페이지 문맥으로 돌아가야 한다.
- 블록 이동과 페이지 전환에 구조 검증 및 실패 사례가 추가된다.
- 페이지가 지나치게 커지면 사용자가 직접 페이지를 나누어야 한다.
- 페이지 내부에 중첩 페이지를 허용하지 않으므로 긴 문서를 여러 페이지로 분리하는 과정에서 콘텐츠 이동이 필요하다.
- 문서 뷰의 범위와 AI가 읽을 정확한 블록 범위를 별도로 설명해야 할 수 있다.

## 검증할 질문

- 사용자는 폴더 블록과 페이지 블록을 생성 시점에 구분해야 하는가?
- 기존 콘텐츠 블록을 새 페이지로 추출하는 동작은 어떤 자손을 함께 이동하는가?
- 페이지를 폴더로 바꾸거나 폴더를 페이지로 바꿀 때 기존 자손을 어떻게 처리하는가?
- 콘텐츠 블록의 직접 URL과 권한은 소유 페이지와 어떻게 표현하는가?
- 보드와 스트림의 범위는 페이지 전체인가, 현재 선택된 콘텐츠 블록의 서브트리인가?
