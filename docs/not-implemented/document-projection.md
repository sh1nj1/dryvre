# 결정론적 문서 투영

> 상태: Backlog — 현재 구현 안함
>
> 이 문서는 향후 구현을 위한 결정 및 권장 기준을 보존한다. 현재 코드·스키마·API 변경을 지시하지 않는다.

## 목적

저장 모델과 온톨로지는 모든 블록이 동일한 순수 블록 모델을 유지한다. `is_page`, `is_root`, `kind` 같은 페이지 구분 컬럼을 추가하지 않고, 선택한 블록을 문서로 표시할 때의 인라인 범위만 결정론적 투영 함수로 정한다.

핵심 원칙은 다음과 같다.

> 문서에 인라인으로 보이는 범위는 트리 구조, `position`, 뷰어 권한, 명시적 `focus`만의 함수다. 뷰포트 높이, 네트워크 청크, 렌더 시간 같은 실행 환경은 투영 결과에 영향을 주지 않는다.

## 후보 데이터 모델

이 설계가 전제하는 최소 코어는 다음 세 테이블이다.

```text
block(id, parent_id, position, content, ...)
grant(...)
ref(...)
```

- `position`은 형제 사이의 fractional rank다.
- 루트는 `parent_id IS NULL`인 일반 블록이다.
- 링크, 태그, 즐겨찾기와 진입점은 `ref`로 표현한다.
- 권한 캐시를 별도로 만들지 않고 `grant`를 기준으로 읽기 가능 여부를 계산한다.

이 모델은 현재 활성 아키텍처와 차이가 있을 수 있으며, backlog 승격 시 별도 정합성 검토가 필요하다.

## 투영 계약

개념적 API는 다음과 같다.

```ts
function projectDocument(
  root: BlockId,
  viewer: ViewerId,
  focus: BlockId[] = [],
): DocNode[]
```

기본 상수는 한 곳에서 공유한다.

```ts
const INLINE_DEPTH = 2;
const INLINE_CHILD_LIMIT = 50;
```

같은 데이터베이스 스냅샷, `root`, `viewer`, `focus`에는 항상 같은 결과를 반환한다. 테스트 가능한 코어에서는 조회된 블록과 읽기 가능 집합 또는 명시적인 스냅샷을 입력으로 받아, 데이터베이스 상태가 숨은 입력이 되지 않게 한다.

서버가 SSR 결과의 권위자가 되며 클라이언트는 뷰포트나 네트워크 상태를 근거로 결과를 다시 자르지 않는다.

## 기본 순회와 컷

- 범위는 `root`의 서브트리다.
- 부모를 먼저 방문하고 형제는 `position` 오름차순으로 방문하는 pre-order를 사용한다.
- 뷰어가 읽을 수 없는 블록은 렌더와 카운트에서 제외한다.
- 상대 깊이가 `INLINE_DEPTH`를 넘으면 재귀를 중단한다.
- 깊이 경계의 블록에 읽을 수 있는 직계 자식이 있으면 depth continuation을 하나 렌더한다.
- 깊이 안에 있더라도 한 부모의 읽을 수 있는 자식이 `INLINE_CHILD_LIMIT`를 넘으면 기본적으로 앞의 50개만 렌더하고 width continuation을 하나 렌더한다.
- continuation은 실제 truncation 경계에서만 나타난다. 모든 블록에 일괄적인 펼치기 chevron을 붙이지 않는다.

continuation 숫자는 뷰어가 읽을 수 있지만 현재 투영에 포함되지 않은 **직계 자식 수**다. 읽을 수 없는 블록의 존재를 숫자로 누출하지 않는다.

## 범위 이동

인라인 아코디언이나 별도의 펼침 상태를 만들지 않는다. 일반 블록과 continuation의 동작은 모두 새 블록 선택으로 귀결한다.

| 대상 | 동작 |
| --- | --- |
| 일반 블록 | 해당 블록을 새 문서 루트로 선택한다. |
| depth continuation | 깊이 경계의 부모 블록을 새 문서 루트로 선택한다. |
| width continuation | 기본 권장안에서는 첫 번째로 생략된 읽기 가능 자식 블록으로 이동한다. |

기존 제안인 “width continuation에서 부모로 다시 진입해 자식 전체를 표시”는 동일한 `INLINE_CHILD_LIMIT`가 다시 적용되어 같은 화면을 반복하므로 채택하지 않는다.

width continuation에 cursor나 pagination을 추가하는 안과 문서 루트의 직계 자식을 제한에서 제외하는 안은 현재 채택하지 않는다. 전자는 선택 외의 투영 상태를 만들고, 후자는 자식 수가 큰 루트에서 렌더 상한을 제거한다.

continuation의 렌더 키와 계측 ID는 배열 인덱스가 아니라 경계로부터 결정한다.

```text
depth:{parentId}
width:{parentId}:{firstOmittedChildId}
```

## Focus 규칙

초기 권장 focus 소스는 검색 결과와 딥링크 타깃이다. 편집 캐럿은 focus에 넣지 않는다. 캐럿 이동 때마다 투영이 바뀌어 문서가 출렁이거나 selection·IME 복원이 불안정해질 수 있기 때문이다. 깊은 블록의 저작은 해당 블록을 새 범위로 선택하도록 유도한다.

focus 후보는 다음 조건을 모두 만족할 때만 유효하다.

- `root`의 descendant다.
- 뷰어가 focus 블록을 읽을 수 있다.
- 렌더할 spine의 각 블록도 뷰어가 읽을 수 있다.

유효하지 않은 focus는 존재 여부를 드러내지 않고 조용히 무시한다. 읽을 수 없는 중간 조상 아래의 읽을 수 있는 자손을 노출할 수 있는지는 권한 의미론을 먼저 확정한 뒤 결정한다.

유효한 focus마다 `root → focus` 경로는 깊이 제한을 무시하고 포함한다. 너비 제한과 충돌할 때 한 부모의 표시 자식은 다음 집합이다.

```text
position 기준 앞 INLINE_CHILD_LIMIT개
∪ 유효한 focus path에 속하는 직계 자식
```

따라서 focus 때문에 표시 수가 50을 넘을 수 있다. continuation 숫자는 전체 읽기 가능 자식 수에서 실제 표시 집합의 수를 빼서 계산한다.

focus 블록의 직계 자식은 무제한으로 강제 포함하지 않는다. 기본적으로 동일한 깊이·너비 규칙을 적용해 fan-out이 큰 블록에서도 상한을 유지한다.

## 루트와 진입점

워크스페이스 루트도 예외 타입으로 취급하지 않고 같은 함수를 적용한다. 자주 진입하는 블록은 새 페이지 타입 대신 `ref` 기반 즐겨찾기 또는 entry point로 노출하는 방향을 우선 검토한다.

## 계측 후보

- continuation 클릭 후 기대한 범위를 찾지 못해 즉시 되돌아가거나 연속 탐색하는 비율
- continuation click-through rate
- 검색 또는 딥링크 진입 시 목표 블록이 최초 서버 투영에 포함된 비율

“범위 예측 오류율”은 실제 이벤트로 측정 가능한 proxy를 먼저 정의해야 한다. 이 지표가 낮게 유지된다면 별도 페이지 온톨로지와 데이터 migration 비용은 정당화하기 어렵다는 가설을 검증한다.

## 승격 시 구현 순서

아래 순서는 향후 이 backlog가 활성 범위로 승격된 뒤에만 사용한다.

1. 순수 projection 코어와 고정 상수, 경계·권한·focus 계약 테스트
2. 서버 projection 및 응답 계약
3. continuation 렌더와 범위 이동 배선
4. 검색·딥링크 focus 배선
5. 계측 이벤트와 지표 정의

## 현재 채택하지 않은 항목

- `is_page`, `is_root`, `kind` 컬럼 또는 별도 Page 엔티티
- 인라인 accordion과 지속되는 펼침 상태
- 뷰포트·네트워크·렌더 시간 기반의 동적 컷
- width continuation용 cursor 또는 pagination
- 문서 루트 직계 자식에 대한 무제한 렌더
- 편집 캐럿을 focus 소스로 사용
- focus 블록의 모든 직계 자식을 무제한 강제 포함

삭제 시 데이터 migration이 없다는 점은 이 접근의 장점이다. 다만 UI, URL, 계측, 클라이언트 계약의 롤백 작업은 남으므로 “롤백 비용 0”이 아니라 **스키마·데이터 migration 비용 0**으로 표현한다.
