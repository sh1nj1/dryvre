# 해커톤 MVP 범위

## 목표 완성도

목표는 기능이 많은 협업 제품이 아니라, 단일 블록 트리가 문서·태스크·대화·AI를 실제로 통합한다는 사실을 3분 안에 입증하는 완성도 높은 수직 슬라이스다.

배포 가능한 demo-grade MVP까지 완성한다. 프로덕션 베타 수준의 실시간 협업, 운영 관리나 모든 장기 설계를 구현하려 하지 않는다.

## 반드시 작동해야 하는 것

| 영역 | 완료 기준 |
| --- | --- |
| 블록 트리 | 생성, 편집, 삭제와 중첩이 저장되고 새로고침 후 유지된다. |
| 문서 뷰 | 같은 블록 트리를 재귀 표시하고 Markdown 본문을 편집한다. |
| 보드 뷰 | `status`가 있는 동일 블록을 상태별로 표시하고 상태를 변경한다. |
| 스트림 뷰 | 선택 블록 아래에 메시지를 자식 블록으로 추가한다. |
| 뷰 전환 | 데이터 변환 없이 같은 ID, 본문, 상태와 선택이 유지된다. |
| 참조 | 다른 블록을 참조하고 링크 또는 backlink로 관계를 확인한다. |
| AI | 현재 서브트리와 참조 블록으로 직접 모델 또는 사람이 선택한 Local Agent를 호출하고 결과를 블록으로 저장한다. |
| Agent loop | PM Agent가 작업을 구체화하고, Developer Agent가 `todo → in_progress → done`으로 닫으며 입력 대기는 `blocked`로 표시한다. |
| 사용자 Inbox | 사용자마다 하나의 Inbox 블록에서 에이전트의 질문과 승인 요청을 스트림으로 받고 답한다. |
| 검색 | 하나의 검색 팝업에서 텍스트와 필터를 조합해 현재 트리를 좁힌다. |
| 데모 UX | seed 데이터, 로딩·오류 처리, 일관된 화면과 공개 실행 URL을 제공한다. |

## 핵심 데모 흐름

1. 문서 뷰에서 출시 블록과 요구사항을 작성한다.
2. 스트림에서 `@PM Agent`에게 실행할 작업을 상세히 정의하도록 요청한다.
3. PM Agent가 결과물, 완료 조건, 제약, 검증 방법과 `@Developer Agent`가 담긴 작업 블록을 만든다.
4. 사용자가 초안을 검토하고 같은 블록을 `todo`로 바꿔 실행을 승인한다.
5. Developer Agent가 완료 조건을 사전 검증한다. 부족한 조건이 있으면 같은 블록을 `blocked`로 바꾸고 사용자의 Inbox에 질문한다.
6. 사용자가 Inbox 스트림에서 필요한 정보를 답하거나 승인을 제공한다.
7. 답변 후 블록이 `todo`로 복귀하고, 조건이 갖춰지면 Developer Agent가 같은 블록을 `in_progress`로 선점해 작업을 시작한다.
8. 결과와 검증 근거를 자식 블록으로 남기고 검증 성공 후 같은 블록을 `done`으로 바꾼다.
9. 문서, 보드와 Inbox 스트림을 오가며 계획, 실행, 대화와 결과가 복사 없이 같은 트리와 참조로 연결됨을 보여준다.

장면별 시간, 화면 동작과 내레이션은 [데모 비디오 시나리오](demo-video-scenario.md)를 따른다. 에이전트의 실행 전 검증과 재개 규칙은 [에이전트 작업 루프](agent-loop-spec.md)를 따른다.

핵심 메시지는 다음과 같다.

> 문서, 업무, 대화와 AI 결과는 서로 변환되거나 복사되지 않는다. 처음부터 모두 같은 블록이다.

Loop Engineering 관점의 보조 메시지는 다음과 같다.

> 사람은 의도와 승인 경계를 정하고, 에이전트는 완료 조건을 검증하며 작업을 끝까지 닫는다. 막히면 추측하지 않고 Inbox로 돌아온다.

## 해커톤에서 구현하지 않는 것

- 완전한 실시간 다중 사용자 편집, 참여 사용자 목록·presence와 충돌 복구 UX
- WebSocket 장애 복구, 멀티 프로세스 fan-out과 운영 모니터링
- 계층적 권한 관리 전체: 권한 단계, 부모-자식 상속, 하위 블록 덮어쓰기, 서버 판정, 설정 UI와 조직 관리
- `op_log` 기반 사용자 실행 취소·재생 화면
- 범용 구독, 읽지 않음 롤업과 이메일·푸시 알림. 에이전트 질문과 승인 요청을 모으는 사용자별 Inbox 블록은 현재 범위다.
- 슬랙 양방향 동기화, OAuth 앱 관리와 외부 MCP 자동 등록
- 패스키, 멀티 프로필, 소셜 로그인과 관리자 설정
- 고급 에디터 플러그인, 표 내보내기, 이미지 편집, Mermaid와 임베드
- 발표 모드, 타이머, 발표자 메모와 슬라이드 동기화
- 캘린더, 회의와 날짜 파싱 연동
- 임의 에이전트 간 위임, 자율 계획 재귀, 장기 세션 모드와 범용 트리거 빌더. PM→사용자 승인→Developer의 고정 작업 루프는 현재 범위다.
- 별도 Plan 엔티티와 타임라인
- Redis, Kafka, Elasticsearch, CRDT와 다중 리전 운영

## 제품에서도 만들지 않는 중복 개념

다음 항목은 단순한 MVP 제외가 아니라 제품 원칙상 제거한다.

- 트리 바깥의 Workspace 엔티티와 워크스페이스 스위처
- Project, Epic, Issue 전용 엔티티나 고정 타입 배지
- 트리 선택과 중복되는 `Subtree` 범위 컨트롤
- 별도 전역 검색과 독립 Filter·Sort 툴바
- 채팅을 위한 별도 Topic·Message 테이블
- 댓글을 문서로 바꾸는 승격·변환 기능
- 링크, 태그와 AI 컨텍스트를 위한 서로 다른 연결 모델

## 목업 주석 규칙

[`demo/mockup/index.html`](../demo/mockup/index.html)에서 붉은 점선과 `OUT OF MVP` 표시는 완성형 제품에는 가능하지만 이번 해커톤에서 구현하지 않는 기능을 뜻한다. 제품 원칙상 영구 제거된 UI는 붉게 남겨두지 말고 목업에서 삭제한다.

## 제출 산출물

- 작동하는 공개 배포 또는 재현 가능한 로컬 실행
- 3분 미만 공개 YouTube 데모
- 프로젝트 설명과 선택 트랙
- 공개 저장소와 라이선스 또는 심사위원에게 공유한 비공개 저장소
- 셋업 및 구현 가이드가 포함된 README
- Codex가 개발에 사용된 방식과 GPT 모델 사용 방식 설명
- 핵심 기능을 개발한 Codex Session ID

## 완료 전 점검

- 신규 사용자가 블록, 트리, 참조 외의 필수 개념을 배워야 하는가?
- 세 뷰가 실제로 같은 블록 ID와 저장 상태를 사용하는가?
- AI 컨텍스트 범위를 화면에서 설명할 수 있는가?
- `todo`가 사용자 승인 경계, `in_progress`가 에이전트 선점, `blocked`가 입력 대기임을 화면에서 이해할 수 있는가?
- 완료 조건이 부족할 때 작업은 `blocked`로 이동하고 질문이 정확한 사용자의 Inbox에 한 번만 생성되는가?
- Inbox 답글 뒤 같은 작업이 재개되고 결과와 검증 근거가 있어야만 `done`이 되는가?
- 네트워크 또는 모델 오류가 데모 전체를 막지 않는가?
- seed 워크스페이스가 제품 주장을 설명하는가?
- README만 보고 새 환경에서 실행할 수 있는가?
- 영상의 모든 장면이 핵심 주장이나 심사 기준을 뒷받침하는가?
