# 에이전트 작업 루프

## 목적

Dryvre의 AI 에이전트는 대화를 생성하는 기능이 아니라, 사람이 정한 의도를 실행 가능한 계약으로 만들고 검증 가능한 결과까지 닫는 참여자다. 이 문서는 해커톤 MVP에서 구현할 하나의 고정된 Loop Engineering 흐름을 정의한다.

```text
의도 → 작업 계약 → 사용자 활성화 → 사전 검증 → 선점 → 실행 → 검증 → 완료
조건 부족·승인 필요 → 사용자 Inbox → 사용자 답변 → 사전 검증 또는 실행 재개
```

PM Agent와 Developer Agent는 별도 작업 데이터 모델을 만들지 않는다. 계획, 작업, 질문, 답변, 결과와 검증 근거는 모두 블록이며 `status`, 부모-자식 관계, 멘션과 `ref`로 연결한다.

## 역할

- 사용자: 목표와 제약을 제공하고, 작업 초안을 `todo`로 바꿔 실행을 허용하며, Inbox에서 질문과 승인 요청에 답한다.
- PM Agent: 현재 블록의 서브트리와 참조를 읽고 실행 가능한 작업 계약을 작성한다. 직접 작업을 활성화하지 않는다.
- Developer Agent: 자신이 명시된 `todo` 작업의 계약을 검증하고 선점한 뒤 실행한다. 근거 없이 완료 처리하지 않는다.

MVP의 역할은 이 두 에이전트로 고정한다. 에이전트가 다른 에이전트를 자유롭게 생성하거나 작업을 재귀 위임하는 범용 조율은 포함하지 않는다.

## 작업 계약

PM Agent가 만드는 작업은 상태 없는 일반 블록이다. 본문이나 자식 블록에 최소한 다음 내용을 사람이 읽을 수 있는 Markdown으로 기록한다.

1. 결과물: 무엇이 바뀌거나 생성되어야 하는가
2. 완료 조건: 어떤 관찰 가능한 상태를 만족해야 하는가
3. 제약: 변경 범위, 금지 사항과 반드시 유지할 사항
4. 검증 방법: 완료를 어떤 검사나 증거로 확인하는가
5. 실행 주체: `@Developer Agent` 멘션

별도 acceptance-criteria 테이블이나 작업 타입을 만들지 않는다. Developer Agent는 이 내용을 구조화해 읽되, 하나라도 실행 또는 완료 판정을 모호하게 만들면 부족한 계약으로 취급한다.

## 활성화와 상태 전이

허용되는 기본 전이는 다음과 같다.

```text
status 없음 --사용자 검토 및 활성화--> todo
todo --계약 충족 + 원자적 선점--> in_progress
in_progress --결과 기록 + 검증 성공--> done
```

- PM Agent의 초안 생성은 실행 승인이 아니다.
- 사용자가 직접 `todo`로 바꾼 작업만 자동 실행 후보가 된다.
- Developer Agent는 작업 또는 작업 계약에 자신이 명시된 경우에만 후보를 처리한다.
- 실행 전 계약이 부족하면 상태를 `todo`로 유지한다. 질문을 보냈다는 이유로 `in_progress`나 `done`으로 바꾸지 않는다.
- 실행 중 사용자 판단이 필요하면 `in_progress`를 유지한 채 기다린다.
- 결과 블록과 검증 근거가 없거나 검증이 실패하면 `done`으로 바꾸지 않는다.

`todo → in_progress`는 작업의 기대 `version`을 포함한 조건부 연산으로 한 번만 성공해야 한다. 동일한 상태 이벤트의 재전달은 같은 실행을 중복 생성하지 않는다.

## 실행 전 검증

Developer Agent는 도구를 실행하거나 외부 상태를 변경하기 전에 다음을 확인한다.

- 요구 결과와 변경 범위가 구체적인가
- 성공과 실패를 구분할 완료 조건이 있는가
- 실행에 필요한 입력, 권한과 사용자 선택이 준비됐는가
- 검증 방법을 실제로 수행할 수 있는가
- 현재 트리의 활성 스펙이나 명시적 제약과 충돌하지 않는가

안전한 읽기와 컨텍스트 확인은 사전 검증에 포함할 수 있다. 파일 변경, 외부 전송, 배포처럼 결과를 바꾸는 행위는 계약이 충족되고 작업을 선점한 이후에만 수행한다.

## 질문과 승인

각 사용자 subject에는 하나의 Inbox 블록을 연결한다. 이 루프에서 요청 대상은 작업을 `todo`로 활성화한 사용자다. 에이전트가 그 사용자의 입력 없이 안전하게 다음 단계로 갈 수 없을 때 Inbox 아래에 스트림 블록을 하나 만든다.

요청 블록에는 다음 정보가 있어야 한다.

- `@사용자` 멘션
- 질문인지 승인 요청인지 구분되는 짧은 제목
- 결정에 필요한 최소한의 맥락과 선택 또는 질문
- 원래 작업 블록에 대한 `ref`
- 응답 전에는 수행하지 않을 행동

사용자는 요청 블록에 답글을 단다. 답글 자체도 블록이며 별도 승인 레코드를 만들지 않는다. 명시적인 승인 문구 또는 질문에 대한 답이 들어오면 대기 중 실행은 해당 답글과 원래 작업을 컨텍스트로 다시 읽는다.

동일한 미해결 조건에는 활성 요청을 하나만 유지한다. 대기 중인 실행은 그 요청 블록 ID를 기록해 재시도나 이벤트 중복으로 같은 Inbox 요청을 반복 생성하지 않는다. 일반 멘션, 구독 알림과 모든 변경 사항을 Inbox로 보내지 않으며, 에이전트가 진행하기 위해 반드시 필요한 질문과 승인만 보낸다.

## 완료와 증거

Developer Agent는 실행 결과를 원래 작업 블록 아래에 편집 가능한 자식 블록으로 기록한다. 최소한 다음을 남긴다.

- 수행한 변경 또는 생성된 결과물
- 실행한 검증과 결과
- 남은 제한이나 후속 작업이 있다면 그 내용

검증이 성공하면 마지막 승인된 연산으로 작업 상태를 `done`으로 바꾼다. 화면의 `done`은 에이전트의 주장만이 아니라 같은 서브트리에서 확인할 수 있는 결과와 증거를 뜻한다.

## 실패와 재개

- 계약 부족: 작업은 `todo`, 실행은 미선점, Inbox 답변 후 계약을 보완하고 사전 검증부터 다시 시작한다.
- 실행 중 승인 대기: 작업은 `in_progress`, 실행은 대기, Inbox 답변 후 같은 실행을 재개한다.
- 검증 실패: 작업은 `in_progress`에 남기고 실패 근거를 기록한다. 사용자 결정이 필요할 때만 Inbox에 요청한다.
- 권한 또는 시스템 오류: 상태를 `done`으로 바꾸지 않고 오류 근거를 기록한다. 자동 재시도는 동일 실행 ID 안에서 멱등성을 보장해야 한다.

## MVP 비범위

- 에이전트의 임의 생성과 에이전트 간 자유 위임
- 목표를 무한히 분해하는 자율 재귀 루프
- 예약 실행, 범용 규칙 빌더와 장기 세션 모드
- 일반 알림 센터, 구독, 읽지 않음 롤업과 이메일·푸시 전송
- 검증 근거 없는 자동 `done` 처리
