# 제품 원칙

## 제품 정의

Dryvre는 일의 배경, 실행 항목, 대화와 AI 결과를 하나의 블록 트리에 보관한다. 같은 블록 집합을 문서, 보드, 스트림으로 다르게 렌더링해 위키·태스크·채팅 사이의 복사와 변환을 없앤다.

사용자가 배워야 할 핵심 개념은 세 개를 넘지 않는다.

1. 블록
2. 트리
3. 참조

별도의 워크스페이스, 프로젝트, 에픽, 이슈, 토픽, 메시지 엔티티를 사용자 개념으로 추가하지 않는다.

## 해결하려는 문제

사람이 협업하려면 최신 정보를 분류하는 문서, 계획과 실행을 관리하는 태스크, 빠르게 소통하는 대화가 필요하다. 기존 제품은 이 세 가지를 별도 데이터 모델로 저장하거나 통합 제품 안에서도 서로 변환한다. 그 결과 같은 내용을 중복 입력하고, 어느 사본이 정본인지 판단해야 하며, AI 컨텍스트도 다시 조립해야 한다.

Dryvre는 정보 블록을 first-class 객체로 삼아 정본을 하나로 유지한다.

## 불변 규칙

### 1. 블록이 유일한 원자 단위다

문단, 제목, 할 일, 이미지, 코드, 메시지와 AI 출력은 모두 동일한 블록이다. 모든 블록은 고유 ID, 부모, 본문, 권한과 참조를 가질 수 있다. 자식을 가진 블록도 별도 종류가 아니라 그냥 블록이다.

### 2. 트리는 하나뿐이다

부모-자식 관계가 유일한 구조다. 채팅이나 태스크를 위해 트리 밖에 두 번째 계층을 만들지 않는다.

트리의 최상위 블록이 곧 사용자가 일하는 공간이다. 별도의 `Workspace` 엔티티나 워크스페이스 스위처를 만들지 않는다.

### 3. 프로젝트·에픽·이슈는 타입이 아니다

워크스페이스, 프로젝트, 에픽, 이슈 같은 구분은 고정된 스키마나 화면 타입이 아니다. 블록의 이름, 속성, 참조와 관계 매핑에 따라 그렇게 해석될 수 있을 뿐이다. 따라서 고정 `Project` 배지나 타입별 전용 컨테이너를 기본 UI에 노출하지 않는다.

### 4. 하나의 트리, 세 개의 뷰

- 문서 뷰: 지식과 정리된 내용을 계층적으로 읽고 편집한다.
- 보드·리스트 뷰: `status`가 있는 블록만 추출해 실행 상태로 묶는다.
- 스트림 뷰: 블록 아래에 생성된 대화 블록을 시간순으로 읽고 쓴다.

뷰 전환은 데이터 이동이나 변환이 아니다. 선택된 블록의 ID, 자식, 참조, 권한과 대화는 그대로 유지된다.

### 5. 채팅은 블록 스트림이다

모든 블록이 대화의 초점이 될 수 있다. 메시지는 현재 블록의 자식이며 토픽도 블록이다. 대화가 정리되면 메시지 블록에 정본 순서 `rank`를 부여해 문서에 보이게 할 뿐, 별도 승격이나 변환을 하지 않는다.

### 6. 참조 하나로 연결 개념을 통합한다

블록 A가 블록 B를 참조하면 그 관계는 동시에 링크, 분류 태그, backlink와 AI 컨텍스트가 된다. 태그, 컨텍스트 칩, 링크된 문서와 본문 링크를 서로 다른 규칙으로 만들지 않는다.

### 7. AI는 블록을 읽고 쓰는 참여자다

AI의 기본 입력 범위는 현재 블록의 서브트리와 명시적으로 참조된 블록이다. 사용자가 별도의 컨텍스트 설정 화면을 배울 필요가 없어야 한다. AI 출력도 편집 가능한 블록으로 트리에 저장한다.

### 8. 에이전트는 검증 가능한 작업 루프를 닫는다

에이전트는 요청에 답하는 데서 끝나지 않는다. 명시적으로 자신이 지정된 `todo` 블록을 발견하면 완료 조건을 먼저 검증하고, 충분할 때만 `in_progress`로 선점해 실행한다. 완료 조건이 부족하거나 사용자 판단이 필요하면 작업을 `blocked`로 바꾸고 질문 또는 승인 요청을 사용자의 Inbox 블록에 남긴다. 답변 후 이전 `todo` 또는 `in_progress` 단계로 복귀한다. 실행 결과와 검증 근거를 같은 작업 블록 아래에 기록한 뒤에만 `done`으로 바꾼다.

상태 변경은 장식이 아니라 사람과 에이전트 사이의 제어 경계다. 상태가 없는 초안은 실행하지 않으며, 사용자가 `todo`로 바꾸는 행위가 실행을 허용하는 명시적 신호다. 상세 계약은 [에이전트 작업 루프](agent-loop-spec.md)를 따른다.

### 9. Inbox도 블록이다

각 사용자는 자신에게 필요한 질문과 승인 요청을 모으는 Inbox 블록을 하나 가진다. Inbox 항목과 답글도 일반 스트림 블록이고 원래 작업은 `ref`로 연결한다. 별도의 알림·티켓·승인 엔티티나 복사본을 만들지 않는다.

### 10. Markdown이 정본이다

`body_md` 한 종류만 저장한다. 에디터는 Markdown과 왕복 손실이 없어야 하며 리치 HTML과 Markdown을 이중 직렬화하지 않는다. 제목, 목록, 코드와 표현 종류는 Markdown 접두사에서 파생하고 문서 구조는 블록 트리가 담당한다.

### 11. 태스크성은 선택적 속성이다

`status`가 `null`인 블록은 태스크가 아니다. 태스크 status는 `todo`, `in_progress`, `blocked`, `done` 네 값만 사용한다. `blocked`는 사람이나 외부 입력을 기다려 진행할 수 없는 상태이며 일반 실행 실패를 뜻하지 않는다. 보드와 진행률은 상태가 있는 블록만 대상으로 한다. 위키나 설명 블록에 `0%` 진행률을 강제하지 않는다.

### 12. 권한은 세 단계이며 상속된다

권한은 읽기, 쓰기, 관리만 둔다. 부모에서 자식으로 상속하고 하위 블록에서 덮어쓸 수 있다. 별도의 권한 캐시나 공유 전파 계층보다 부모 체인 조회를 우선한다.

## 피해야 할 부패 패턴

- 링크, 태그, 컨텍스트와 본문 링크를 서로 다른 연결 모델로 구현하는 것
- 블록 본문을 주소나 권한이 없는 하나의 리치텍스트 덩어리로 저장하는 것
- 단순한 실행 루프를 범용 에이전트 조율 프레임워크, 자율 세션 모드나 숨은 트리거 규칙으로 확장하는 것
- 권한 단계를 늘리고 캐시 무효화 문제를 만드는 것
- 모든 노드에 진행률을 표시하는 것
- 완료, 아카이브, 삭제와 필터가 서로 예측 불가능하게 겹치는 것
- 채팅을 트리 밖의 토픽·메시지 데이터 모델로 만드는 것
- Inbox를 트리 밖의 알림·승인 데이터 모델로 만드는 것
- 같은 범위를 워크스페이스 스위처, 트리 선택, 서브트리 버튼으로 중복 제어하는 것

기능을 추가할 때 사용자가 배워야 할 새 개념이나 별도 엔티티가 생긴다면 먼저 기존의 블록, 트리, 참조로 표현할 수 있는지 검토한다.
